# 录音延迟播放设计

## 目标与范围

网页删除本地音频文件选择和上传入口，只保留麦克风录音、试听和上传。用户先选择“立即播放”或“延迟播放”；选择延迟播放后才显示秒数输入，允许 `1–604800` 秒（最多 7 天），立即播放提交 `0` 秒。

## 关键方案

Cloudflare Pages Function 以服务器当前时间加延迟秒数计算 `playAt`，并把该 UTC 毫秒时间写入 R2 对象 key：`incoming/<playAt>-<uploadId>-recording.<ext>`。消费者解析 key，只从已到播放时间的对象中选择最早一条；旧格式对象无法解析计划时间时按立即播放处理，保持向后兼容。

选择对象 key 而非浏览器倒计时或额外数据库，是因为计划可跨手机关闭、网络中断和树莓派重启保留；相比逐个读取 R2 元数据，也减少轮询请求。当前单消费者模型适合少量用户和一台播放设备。

## 数据流与异常

1. 浏览器完成录音，用户输入延迟秒数并点击上传。
2. 前端校验整数范围，将 `file` 和 `delaySeconds` 作为表单提交。
3. 服务端再次校验，计算 `playAt` 并写入 R2。
4. 消费者轮询 `incoming/`，跳过未到期对象，下载并播放最早到期对象。
5. 播放成功移动到 `played/`，播放失败移动到 `failed/`，下载失败保留原对象重试。

非法延迟返回 `INVALID_DELAY`。消费者遇到旧 key 或畸形计划时间时立即处理，避免对象永久卡住。

## 页面设计

视觉基调保持当前温暖纸张色与编辑式大标题。录音区成为唯一主操作，试听下方提供“立即播放 / 延迟播放”二选一；仅在选择延迟播放后显示秒数输入和预计等待提示，不增加额外页面或弹窗。中英文切换继续覆盖全部文案。
